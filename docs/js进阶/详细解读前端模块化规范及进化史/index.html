详细解读前端模块化规范及进化史

面试题：重写reduce方法

Array.prototype.reduce = function(callbackFn,initialvalue){
    var self = this,
        result = initialvalue,
        index = 0
    if (typeof callbackFn !== 'function') return new TypeError('callbackFn is not a function')
    if (typeof initialvalue === 'undefined') { 
        result = self[0]
        index = 1
    }
    //迭代数组
    for (; index < self.length; index++) { 
        result = callbackFn(result,self[index],index)
    }
    return result
}

let arr = [10, 20, 30, 40, 50]
let result = arr.reduce((x, item) => x + item)
console.log(result);

最开始的模块化开发：把每个不同模块的代码放在不同的文件中，最后在页面分别导入

- 需要自己"手动分析出依赖"，规划出导入的先后顺序[麻烦]：即便基于grunt/gulp/webpack等处理，也需要知道依赖关系，按照依赖关系打包...
- 如果不基于闭包把每个模块中的代码私有化处理，最后处理起来的时候，容易引发"全局变量污染"
解决私有化：自执行函数执行，产生闭包即可
解决模块之间的相互访问：
- 把需要供外面访问的内容，暴露到全局上  window.xxx = xxx,但是这种方式在暴露更多的方法的时候，也可能会导致全局变量的冲突
- 把模块中需要暴露的属性方法放在一个对象中管理，最后基于模块名存储这个对象即可
```js
let xModule = function(){
    ...

    return {
        //包含需要暴露给外面的属性方法
        fn，
    }
}
访问：xModule.fn()
```
总结：这种处理方案，即保证了模块代码间的私有化，也支持模块间的相互访问，而且避免了全局变量的污染，我们把这种代码设计方法称之为"单例设计模式"，所有设计模式其实都是一种思想，着这种思想解决了莫=某一类问题!

AMD设计思想:在单例设计模式的基础上，有效的解决了模块之间的依赖问题，告别之前手动一点点分析依赖，按照顺序依次导入问题了；而且可以结合gulp/grunt等，最后把各个模块代码合并压缩打包在一起...

- 但是依赖的模块需要"前置导入"

commonJS模块规范 [模块的导入和导入]：Node自带的模块规范（浏览器不支持）
定义模块：创建的每一个js文件，就是定义一个单独的模块

- 导出模块的方法：
```js
module.exports = {
//包含需要供外部调用的属性和方法
}
```
- 导入指定的模块：
`const x = require('模块地址，导入自己的模块需要加"./"')`
基于x接收导入的对象，后期基于 x.xxx 访问即可！！

CommonJS模块导入是"按需"的。随时用随时导入即可。不想AMD都需要前置处理

tip

CommonJS规范比CMD用起来更加简单，从导入机制等原理上也比AMD性能高一些 但是CommonJS不支持浏览器端 所以 淘宝玉伯 写了一个插件 sea.js [把其定义为CMD模块规范] 本质：把CommonJS规范搬到浏览器端运行

再后来 ES6本身就提供了更好用的模块规范：ES6Module,sea.js代表的CMD规范就被Pass掉了

- 在浏览器端开启ES6Module规范
    - 基于标准的http/https协议的web服务预览页面
- 定义模块：和commonJS类似，创建一个JS就相当于创建一个模块
导出/导入模块：
    方案一：
    export default
语法：
    export 声明变量且赋值；
    export default 值： -> 在一个模块中只能使用一次
    每个模块导入一个Module对象
    {
        num:10,
        ...
        default:sum
    }
    import x from '模块地址'; 
        - 浏览器端使用，地址中模块的后缀不能省略
        - 只能接受到基于 export default 导出的这个值
        - 原理：找到导出Module对象中的default属性值，把属性值赋值给x变量
        - 但是不能在这直接给x解构赋值， 例如：import {x，y} from '模块地址'； 这样是不能给default后面的值解构赋值
        ，则先基于x接收，然后再给x解构赋值即可，例如：const {n,m} = x;
    
    import * as x from '模块地址' 把当前模块导出所有内容获取到，赋值给x变量，后期基于 x.xxx 访问即可[含：x.default导出的值]

    import {num,obj} from '模块地址'
    直接解构赋值，是把模块导出的Module中所有内容(不含default)进行解构赋值
